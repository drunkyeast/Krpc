# 我的README

## 感觉rpc与vpn很像啊(后面要实现一个downloader小工具)
RPC和VPN在设计思想上有着惊人的相似之处，它们都采用了透明代理模式来隐藏底层的复杂性。
VPN工作在网络层，通过创建加密隧道让远程网络看起来像本地网络，使得所有应用程序都能无感知地访问远程资源；
RPC工作在应用层，通过封装网络传输细节让远程函数调用看起来像本地函数调用，使得开发者无需关心序列化、网络通信等底层实现。
利用RPC也可以像VPN一样绕过国内的网络限制, 比如下载某些东西卡了, 我会ssh到日本的服务器, 然后wget/curl下载, 最后再scp传回来. 可以用RPC来实现一个小的downloader小应用来实现, 代替ssh+wget+scp多步操作.
这是RPC的一个边缘的小应用. RPC主要用途: 微服务架构中的服务间通信(90%), 例如: 用户服务 ←RPC→ 订单服务 ←RPC→ 支付服务 ←RPC→ 库存服务. 分布式计算(5%), 其他如下载等等(5%).

## 根据施磊的话来梳理一下, 根据Kserver.cc的main函数
服务提供方通过rpcprovider注册服务对象Service和服务方法MethodDescriptor, 记录到一个map表中.
provider.run()启动后, 利用muduo启动了一个epoll+多线程的服务器(或者可以说多Reactor多线程), 这里面绑定了IP+Port, 然后就可以接受远程连接了
远程有新连接过来, 就会启动OnConnection方法回调了, 有新消息就是OnMessage回调.
OnMessage逻辑很重要, 我写了详细的注释. 解析参数 + 反序列化到request + 调用对应的本地业务方法 + 最后再回调 
回调里面的逻辑会对response进行序列化然后用用muduo的send发送给客户端. 最后可以选择是否shutdown(长连接短连接嘛)

OnMessage是核心回调, 完成了将本地方法变成可远程调用的方法. 借助了protobuf和muduo库.如果不用这两个工具, 那要写半天.
RpcProvider是核心类, 干了三件事: muduo网络功能, protobuf序列化和反序列化, 注册服务到网络或者说是一个路由功能--让rpc请求知道调用哪一个服务的那一个方法

## 然后是日志模块了(施磊老师讲得真好) 但不是重点啊, 可略.
首先不能总是打印到终端, 输出到一个log.txt文件中, 来排查错误. 客户端和服务器都需要日志.
日志是一个磁盘的IO, 不能RPC的客户端和服务端做磁盘IO, 影响RPC的速度了. 
所以RPC把信息写到内存中, 用一个"消息队列", 让一个日志线程来做磁盘IO.
因为涉及多线程, 要保证线程安全.
然后是那个"消息队列"——kafka. zeromq（或者叫消息中间件）但zeromq只在内存中传递,不涉及磁盘.
本项目用的glog, 自动实现 线程安全和消息队列磁盘IO.
本项目KrpcLogger只有.h文件没有.cpp, 且仅仅做了一个glog的初始化, 然后LOG调用也用的是glog的宏, 没用到KrpcLogger里面的函数.
核心：glog 已经是一个生产级的异步日志库，内部有完善的线程、缓冲、批量 IO 机制，你直接用就行，不需要自己实现！这就是为什么大家都用 glog，而不是自己写日志系统。

## 为什么要做c++版的rpc？

1.高性能需求

c++以其高效的内存管理和底层控制能力，成为性能要求比较高的系统(如金融、游戏服务器、实时通信系统)的首选语言。

在这些场景下，RPC框架需要尽可能减少通信开销，而C++天生的性能优势可以满足这一需要求。


2.系统级开发

很多底层基础设施(如数据库、中间件、分布式存储系统)都是用c++开发。

这些系统需要一个与语言无缝结合的高效RPC框架，避免因语言间的切换导致性能损耗


3.跨平台

C++的可移植性在不同平台(如linux、Windows、嵌入式系统)上广泛使用。

一个C++RPC框架能够为这些多平台环境提供统一的通信接口，降低开发成本。

4.灵活性与可扩展性

与某些语言的封闭生态不同，C++允许开发者灵活地调整底层实现。例如：可以定制序列化协议(如Protobuf、Thrift)、网络传输方式(如TCP、UDP、QUIC)等，以满足不同场景的需求。

关于C++版RPC框架的使用场景：

* 微服务架构：在微服务架构中，服务通常分布在不同的网络和不同的服务器上，此时就需要一个高效的通信手段就是我们的rpc。
* 实时通信：如在线游戏、视频直播、即时通信等场景，要求低延迟和高吞吐。C++RPC可以通过优化网络传输协议和序列化协议，提供实时性保障。
* 分布式存储与计算：像hadoop、或者你做个raft的共识算法的话，也可也发现我们在不同节点之间使用rpc传递数据进行通信。
* 嵌入式系统： 在嵌入式设备之间的通信中，资源有限且性能要求严格。C++的轻量级特性使其成为嵌入式RPC实现的理想选择。
* 跨语言调用： C++ RPC框架通常支持多语言绑定（如Python、Java），可以用作跨语言调用的桥梁。例如，在后端服务使用C++开发的情况下，前端服务可以通过RPC框架调用其功能。
